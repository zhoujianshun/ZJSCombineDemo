# 第1章：Combine 基础概念

> 对应代码：`Chapter1_Basics/BasicsViewController.swift`

Combine 的核心只有三个角色：**Publisher**、**Subscriber** 和 **Subscription**。理解它们之间的关系是学习 Combine 的基础。

---

## 1.1 三者的关系

```
┌───────────┐    subscribe    ┌────────────┐
│ Publisher  │ ◄───────────── │ Subscriber │
└─────┬─────┘                └──────┬─────┘
      │                             │
      │  receive(subscription:)     │
      │ ──────────────────────────► │
      │                             │
      │     request(.max(N))        │
      │ ◄────────────────────────── │
      │                             │
      │     receive(value)          │
      │ ──────────────────────────► │  (可重复 N 次)
      │                             │
      │  receive(completion)        │
      │ ──────────────────────────► │
      └─────────────────────────────┘
```

1. **Subscriber** 向 **Publisher** 发起订阅
2. **Publisher** 创建 **Subscription** 并交给 Subscriber
3. **Subscriber** 通过 Subscription 请求数据（Demand）
4. **Publisher** 按需发送值，最终发送完成或错误

---

## 1.2 Publisher（发布者）

Publisher 是一个**协议**，定义了两个关联类型：

```swift
protocol Publisher {
    associatedtype Output   // 发出的值的类型
    associatedtype Failure: Error  // 可能的错误类型（Never 表示不会出错）
}
```

### 内置 Publisher 类型

#### Just

发出**单个值**后立即完成。最简单的 Publisher。

```swift
Just(42)  // Output = Int, Failure = Never
```

**使用场景**：提供默认值、在 `catch` 中返回备用值、测试时模拟数据。

#### Future

**异步**产生**单个值**的 Publisher。通过 `promise` 回调传递结果。

```swift
Future<String, Never> { promise in
    DispatchQueue.main.asyncAfter(deadline: .now() + 1) {
        promise(.success("异步结果"))
    }
}
```

**注意**：Future 在**创建时就开始执行**闭包（eager），而不是等到订阅时。如果需要延迟执行，用 `Deferred` 包裹。

#### Empty

不发出任何值，直接完成（或永不完成）。

```swift
Empty<Int, Never>()                    // 立即完成
Empty<Int, Never>(completeImmediately: false)  // 永不完成
```

**使用场景**：在条件分支中需要一个"什么都不做"的 Publisher。

#### Fail

立即发出一个错误。

```swift
Fail<String, MyError>(error: .networkError)
```

**使用场景**：测试错误处理链路、条件判断后直接返回错误。

#### Deferred

**延迟**创建 Publisher——每次有新的 Subscriber 订阅时，才执行闭包创建 Publisher。

```swift
var count = 0
let deferred = Deferred {
    count += 1
    return Just("第 \(count) 次")
}
// 每次订阅都会重新执行闭包，count 递增
```

**使用场景**：需要每次订阅产生不同结果（如每次重新发起网络请求）。

## 1.3 Subscriber（订阅者）

Subscriber 是接收 Publisher 发出值的一方。日常开发中很少自定义 Subscriber，主要使用以下两个内置方法：

### sink

最常用的订阅方式，通过闭包接收值和完成事件。

```swift
publisher.sink(
    receiveCompletion: { completion in
        // 处理完成或错误
    },
    receiveValue: { value in
        // 处理每个值
    }
)
```

如果 Publisher 的 Failure 是 `Never`（不会出错），可以省略 completion：

```swift
publisher.sink { value in
    // 只处理值
}
```

### assign

将 Publisher 发出的值**直接赋给**某个对象的属性。要求 Failure 为 Never。

```swift
publisher.assign(to: \.text, on: myLabel)
```

还有一种更现代的用法，配合 `@Published` 属性：

```swift
publisher.assign(to: &$myPublishedProperty)
```

这种形式不返回 AnyCancellable，由系统自动管理生命周期。

## 1.4 Subscription 与 AnyCancellable

### 为什么需要存储订阅？

`sink` 和 `assign` 返回的是 `AnyCancellable` 对象。如果没有保存它，订阅会**立即被取消**：

```swift
// ❌ 错误：cancellable 没有存储，sink 不会收到任何异步值
publisher.sink { value in print(value) }

// ✅ 正确：存储到 Set 中
publisher.sink { value in print(value) }
    .store(in: &cancellables)
```

### 生命周期管理的最佳实践

```swift
class MyViewController: UIViewController {
    // 在类中声明 Set<AnyCancellable>
    private var cancellables = Set<AnyCancellable>()

    func setupBindings() {
        publisher
            .sink { [weak self] value in  // 用 [weak self] 避免循环引用
                self?.updateUI(value)
            }
            .store(in: &cancellables)  // 存储订阅
    }
    // ViewController 销毁时，cancellables 被释放，所有订阅自动取消
}
```

### 手动取消

```swift
let cancellable = publisher.sink { ... }
cancellable.cancel()  // 手动取消，之后不再收到值
```

## 1.5 关键概念总结

| 概念 | 角色 | 类比 |
|------|------|------|
| Publisher | 数据源 | 报纸出版社 |
| Subscriber | 数据消费者 | 报纸读者 |
| Subscription | 订阅凭证 | 订阅合同 |
| AnyCancellable | 可取消的订阅句柄 | 退订按钮 |
| Demand | 请求的数据量 | "我要 N 份报纸" |

> **重要**：一定要理解 AnyCancellable 的生命周期。这是初学 Combine 最常遇到的问题——"为什么我的 sink 收不到值？"答案往往是忘了 `.store(in:)`。
