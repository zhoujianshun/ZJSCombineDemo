## 8.1 自定义 Publisher（操作符组合法）

虽然 Combine 提供了 `Publisher` 协议让你从零自定义，但实践中更推荐**组合已有操作符**来创建自定义 Publisher。

### 示例：倒计时 Publisher

```swift
func countdown(from count: Int, interval: TimeInterval = 1.0) -> AnyPublisher<Int, Never> {
    Timer.publish(every: interval, on: .main, in: .common)
        .autoconnect()
        .scan(count) { current, _ in current - 1 }
        .prefix(while: { $0 >= 0 })
        .eraseToAnyPublisher()
}

// 使用
countdown(from: 5)
    .sink(
        receiveCompletion: { _ in print("倒计时结束！") },
        receiveValue: { print("⏱ \($0)") }
    )
// 输出: 4, 3, 2, 1, 0, 倒计时结束！
```

**组合思路**：
1. `Timer.publish` — 提供时间驱动
2. `.scan` — 每次 tick 递减计数
3. `.prefix(while:)` — 到 0 时自动完成

### 示例：工厂函数（模拟 Rx 的 create）

用 `Deferred` + `Future` 封装任意异步操作为 Publisher：

```swift
func fetchTemperature(city: String) -> AnyPublisher<Double, Error> {
    Deferred {
        Future { promise in
            // 模拟异步 API 调用
            DispatchQueue.global().asyncAfter(deadline: .now() + 1) {
                let temp = Double.random(in: -10...40)
                promise(.success(temp))
            }
        }
    }
    .eraseToAnyPublisher()
}
```

**为什么用 Deferred 包裹 Future？**
- `Future` 在**创建时**就执行闭包（eager）
- `Deferred` 让它变成**订阅时**才执行（lazy）
- 这样每次订阅都会重新发起请求

## 8.2 背压（Backpressure）

> 对应代码：`AdvancedTopicsVC.swift` + `OneByOneSubscriber.swift` + `BatchSubscriber.swift`

### 为什么需要背压？

想象一个场景：Publisher 每毫秒产生一个值，但 Subscriber 需要 100ms 才能处理一个值。如果没有流量控制，Subscriber 的缓冲区会不断积压，最终内存爆炸。

**背压就是 Subscriber 告诉 Publisher "我能处理多少数据" 的机制**，防止生产速度远超消费速度。

### 生活类比

```
🍔 餐厅类比：

没有背压的厨房：
  服务员: "所有菜单一起下！"
  厨师:    同时做 50 道菜 → 忙不过来 → 菜凉了 → 出错

有背压的厨房：
  服务员: "先做 3 道"              ← request(.max(3))
  厨师:    做完 3 道，端出去
  服务员: "好了，再来 3 道"        ← 返回 .max(3)
  厨师:    继续做...
  
结果: 每道菜都是热乎的，厨房不会爆单
```

### 背压的底层协议

Subscriber 协议中有两个方法控制背压：

```swift
protocol Subscriber {
    // 1️⃣ 收到订阅时被调用，通过 subscription 发出初始请求量
    func receive(subscription: Subscription)
    //    → 在这里调用 subscription.request(.max(N)) 请求 N 个值

    // 2️⃣ 每收到一个值时被调用，返回值决定"还要几个"
    func receive(_ input: Input) -> Subscribers.Demand
    //    → 返回 .max(M) 表示"再给我 M 个"
    //    → 返回 .none   表示"暂时不要了"

    // 3️⃣ 收到完成/错误
    func receive(completion: Subscribers.Completion<Failure>)
}
```

### Demand 类型

| Demand | 含义 | 使用场景 |
|--------|------|---------|
| `.unlimited` | 把你有的全给我 | `sink` 默认使用，适合数据量可控的场景 |
| `.max(N)` | 再给我 N 个 | 分页加载、批量处理 |
| `.none` | 暂时不要了 | 本批还没处理完，等等 |

**重要**：Demand 是**累加**的，不能减少。比如先 `request(.max(3))`，再返回 `.max(2)`，总共可接收 5 个值。

### 示例1: sink 的默认行为（unlimited）

```swift
(1...5).publisher
    .print("📊")
    .sink { value in print("收到: \(value)") }
```

Xcode 控制台输出：

```
📊: receive subscription: (1...5)
📊: request unlimited              ← 一次请求全部
📊: receive value: (1)             ← 值一股脑全发过来
📊: receive value: (2)
📊: receive value: (3)
📊: receive value: (4)
📊: receive value: (5)
📊: receive finished
```

`sink` 在底层调用了 `subscription.request(.unlimited)`，相当于"你有多少给多少"。

### 示例2: 自定义 Subscriber — 逐个请求

```swift
class OneByOneSubscriber: Subscriber {
    typealias Input = Int
    typealias Failure = Never

    func receive(subscription: Subscription) {
        subscription.request(.max(1))       // 初始只要 1 个
    }

    func receive(_ input: Int) -> Subscribers.Demand {
        print("处理: \(input)")
        return .max(1)                      // 每处理完 1 个，再要 1 个
    }

    func receive(completion: Subscribers.Completion<Never>) {
        print("完成")
    }
}
```

运行效果：

```
📩 收到 Subscription
📤 request(.max(1)) → 先要 1 个

📥 收到值: 1
   处理中...
📤 返回 .max(1) → 再要 1 个

📥 收到值: 2
   处理中...
📤 返回 .max(1) → 再要 1 个

... (逐个进行，Subscriber 永远不会被淹没)

✅ 全部完成
```

### 示例3: 自定义 Subscriber — 分批请求

```swift
class BatchSubscriber: Subscriber {
    let batchSize = 2
    var receivedInBatch = 0

    func receive(subscription: Subscription) {
        subscription.request(.max(batchSize))   // 第1批: 要 2 个
    }

    func receive(_ input: Int) -> Subscribers.Demand {
        receivedInBatch += 1
        if receivedInBatch >= batchSize {
            receivedInBatch = 0
            return .max(batchSize)              // 本批处理完，要下一批
        }
        return .none                            // 本批还没收满，不追加
    }
}
```

运行效果（输入 1...7）：

```
📤 第1批: request(.max(2))
📥 收到: 1  (本批 1/2)
📥 收到: 2  (本批 2/2)  ✅ 本批完毕! → 请求下一批

📥 收到: 3  (本批 1/2)
📥 收到: 4  (本批 2/2)  ✅ 本批完毕! → 请求下一批

📥 收到: 5  (本批 1/2)
📥 收到: 6  (本批 2/2)  ✅ 本批完毕! → 请求下一批

📥 收到: 7  (本批 1/2)
✅ 全部完成，共 7 个值
```

### 实际应用场景

| 场景 | 背压策略 | 说明 |
|------|---------|------|
| 普通 UI 绑定 | `.unlimited` (sink) | 数据量小，直接全部接收 |
| 分页加载列表 | `.max(pageSize)` | 每页加载 20 条，滚到底再请求下一页 |
| 数据库批量写入 | `.max(batchSize)` | 每批写 100 条，写完再请求下一批 |
| 高频传感器数据 | `.max(1)` + throttle | 逐个处理，配合节流避免积压 |
| 大文件逐行处理 | `.max(bufferSize)` | 按缓冲区大小分块处理 |

### 日常开发需要关心背压吗？

**大多数时候不需要**。`sink` 和 `assign` 使用 `.unlimited`，对于网络请求、UI 事件等场景完全够用。

只有当你遇到以下情况时才需要考虑：
1. Publisher 产生值的速度远超 Subscriber 处理速度
2. 需要精确控制数据接收节奏（如分页、批处理）
3. 内存敏感的场景（处理大量数据流）

Combine 内置的 `buffer`、`throttle`、`debounce` 操作符已经在操作符层面解决了大部分背压问题。自定义 Subscriber 是最底层的控制手段。

## 8.3 Combine ↔ async/await 桥接

iOS 15 引入了 `AsyncSequence` 桥接，让 Combine 和 async/await 可以互相转换。

### Publisher → async/await

使用 Publisher 的 `.values` 属性，将其转为 `AsyncSequence`：

```swift
// Publisher → AsyncSequence → for-await-in
Task {
    for await value in [1, 2, 3, 4, 5].publisher.values {
        print("async: \(value)")
    }
    print("完成")
}
```

其他转换方式：

```swift
// 获取第一个值
let first = await somePublisher.values.first(where: { _ in true })

// 获取所有值（收集到数组）
var results: [Int] = []
for await value in publisher.values {
    results.append(value)
}
```

### async/await → Publisher

用 `Deferred` + `Future` + `Task` 将 async 函数包装为 Publisher：

```swift
func asyncFunction() async -> String {
    try? await Task.sleep(nanoseconds: 1_000_000_000)
    return "async result"
}

// 包装为 Publisher
let publisher = Deferred {
    Future<String, Never> { promise in
        Task {
            let result = await asyncFunction()
            promise(.success(result))
        }
    }
}
```

### 何时用 Combine？何时用 async/await？

| 场景 | 推荐方案 | 原因 |
|------|---------|------|
| 单次异步操作 | async/await | 语法简洁 |
| 持续的事件流 | Combine | Publisher 天然支持多值 |
| 多个事件的组合/变换 | Combine | 操作符丰富 |
| UI 数据绑定 | Combine + @Published | 与 UIKit/SwiftUI 集成好 |
| 简单的网络请求 | async/await | 线性代码更易读 |
| 复杂的请求链/重试/防抖 | Combine | 操作符链式调用 |

## 8.4 类型擦除：eraseToAnyPublisher

### 为什么需要类型擦除？

Combine 操作符会产生**深度嵌套的泛型类型**：

```swift
// 不擦除时的真实类型（极其冗长）
let publisher: Publishers.Map<Publishers.Filter<Publishers.Sequence<[Int], Never>>, String>
```

使用 `eraseToAnyPublisher()` 简化为：

```swift
let publisher: AnyPublisher<String, Never>
```

### 何时使用？

1. **函数返回值**——对外暴露简洁的类型

```swift
func searchUsers(query: String) -> AnyPublisher<[User], Error> {
    URLSession.shared.dataTaskPublisher(for: makeURL(query))
        .map(\.data)
        .decode(type: [User].self, decoder: JSONDecoder())
        .eraseToAnyPublisher()  // ← 隐藏内部实现细节
}
```

2. **存储为属性**——避免泛型类型暴露

```swift
class ViewModel {
    var isFormValid: AnyPublisher<Bool, Never> {
        // 复杂的组合逻辑...
        .eraseToAnyPublisher()
    }
}
```

3. **条件分支返回不同 Publisher**

```swift
func makePublisher(useCache: Bool) -> AnyPublisher<Data, Error> {
    if useCache {
        return Just(cachedData)
            .setFailureType(to: Error.self)
            .eraseToAnyPublisher()
    } else {
        return URLSession.shared.dataTaskPublisher(for: url)
            .map(\.data)
            .mapError { $0 as Error }
            .eraseToAnyPublisher()
    }
}
```
